# Summary

Agents can advertise user-invocable skills (slash commands) over their existing gRPC stream; the web playground surfaces them in a `/` popover and forwards a typed `SkillInvocation` to the agent when the user picks one. No new RPCs, no new connections, no static manifest — the catalog is built dynamically from `AddSkill` / `RemoveSkill` messages the agent sends, and dies with the process.

# Design

- **`proto/astro/messaging/v1/skill.proto` (new)** — `Skill { name, description, long_description }`, `AddSkill`, `RemoveSkill`, `SkillInvocation { skill_name, args }`. `name` is the unique key; re-adding replaces; removing an unknown name is a no-op.
- **`ConversationRequest` oneof** gains `AddSkill`, `RemoveSkill` (agent → server) and `SkillInvocation` (server → agent forwarding). **`AgentResponse` oneof** gains `SkillInvocation`. All field numbers are additive — old generated code on either side keeps deserializing.
- **`internal/store.SkillsStore`** — in-memory `map[name]*Skill` behind a `sync.RWMutex`. Methods: `Add` (drops nil / empty-name to keep the catalog addressable), `Remove`, `Get`, `List` (sorted by name for stable UI ordering). Lifetime is the messaging-server process; agents re-register on stream reconnect.
- **`internal/grpc/server.go`** — `ProcessConversation` loop routes `AddSkill` / `RemoveSkill` into the store. New `HandleSkillInvocation(ctx, conv, *SkillInvocation)` mirrors `HandleIncomingMessage`: finds the agent's stream, wraps the invocation in `AgentResponse{skill_invocation: …}`, sends it.
- **`internal/adapter.SkillInvoker` (new interface)** — one method, implemented by the gRPC server, consumed by the web adapter. Keeps the web → gRPC dependency one-way and mockable.
- **Web adapter** — `GET /api/skills` returns the catalog (`{ skills: [{ name, description, longDescription? }, …] }`, empty array when empty — never 404). `POST /api/conversations/{id}/invocations` accepts `{ skill, args? }`, validates `skill` against the store (404 if unknown — a misbehaving client can't make the agent see arbitrary slash commands), forwards via `SkillInvoker`. 202 on accept, 503 if no agent stream is attached.
- **Server wiring** — one `SkillsStore` constructed in `cmd/server/main.go`, threaded through `grpc.NewServer` and `initializeAdapters`; web adapter wires the gRPC server as its `SkillInvoker` after both exist (same DI pattern as `AudioForwarder`).

# Agent surface

| SDK | Outbound | Inbound |
|---|---|---|
| Go (`pkg/client`) | `stream.AddSkill(*pb.Skill)` / `RemoveSkill(name)` | `Receive` → `AgentResponse.SkillInvocation` oneof |
| Node (`sdk/node`) | `stream.addSkill(skill)` / `removeSkill(name)` | `stream.on('skillInvocation', …)` event + `AgentResponse.skillInvocation` |
| Python (`sdk/python`) | `add_skill(skill)` / `remove_skill(name)` helpers (build `ConversationRequest`, caller sends) | `AgentResponse.skill_invocation` |

No SDK exposes a higher-level "skill registry" abstraction — the agent owns the policy of when to add/remove; the SDK only saves it from remembering the oneof layout.

# Compatibility

- **Slack / Discord adapters**: untouched. They never see `AddSkill` / `RemoveSkill` / `SkillInvocation` and never get asked for skills. The web adapter is the only skills surface.
- **Existing agents**: opt-in. An agent that never calls `addSkill` keeps the playground's `/` as a normal character, sees no `SkillInvocation` payloads, and is unaffected.
- **Proto wire**: purely additive (new field numbers, new file, no renames or type changes). Unknown-field skipping in proto3 keeps mixed old/new pairings working.

# Migration

None for existing deployments. To use the feature:

1. Regenerate protos (`moon run messaging:proto-gen`).
2. Rebuild server + SDK consumers.
3. In the agent, call `addSkill` after the gRPC stream connects and handle the `skillInvocation` event/oneof.

Restarts wipe the catalog by design — agents re-register on reconnect, and the playground catalog auto-recovers because the stream re-attaches.
